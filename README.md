# CS-230-Operating-Platforms

## Briefly summarize The Gaming Room client and their software requirements.
The client was "The Gaming Room," and they wanted to expand their existing Android-only game, "Draw It or Lose It," to other platforms. My task was to design a system architecture that allowed for this expansion, specifically using a client-server architectural pattern to ensure the game could eventually support web, mobile, and console clients using a shared REST API.
## What did you do particularly well in developing this documentation?
I think what I did well was understanding the design constraints around developing a web-based application with a client-server architecture. Figuring out what the advantages and disadvantages are, and where we could bring out the biggest wins with minimal drawbacks.
## What about the process of working through a design document did you find helpful when developing the code?
Working through the design document forces you to think through the problem and consider trade-offs before committing to a solution. By the time I would start coding, I had already vetted the requirements, identified potential hurdles, and spotted most of the pitfalls, which would make the implementation process much cleaner.
## If you could choose one part of your work on these documents to revise, what would you pick? How would you improve it?
The evaluation section of the document could be structured better, especially for the "client" and "development tools" piece. It was a little redundant and unnecessary to compare and contrast the different operating systems since the approach was a web-based application, and I ended up repeating myself because web code is nearly the same across OSs. I think I would just focus on the server trade-offs instead.
## How did you interpret the user’s needs and implement them into your software design? Why is it so important to consider the user’s needs when designing?
I interpreted the user's needs by really putting myself in the user's shoes. The client already has a successful version of the game, so how do I meet the client's request without degrading the user's experience? Keeping the game consistent, responsive, and fun regardless of which device they were using is basically the core user need. Keeping that in mind helped me avoid over-engineering or worse, solving the wrong problem. It's so important to consider the user's needs when designing because it's easy to get caught up in the technical requirements, budgets, and timelines between client and contractor, and accidentally lose the plot for the ones actually using the software. 
## How did you approach designing software? What techniques or strategies would you use in the future to analyze and design a similar software application?
I approached this project by focusing on the client's requirements, which are to expand this to multiple platforms under a reasonable budget. Therefore, by focusing on a client-server split architecture, I was able to isolate the backend logic, the stuff that lives on the OS, from the specific client the user might be on, the "devices and their web platforms". Something I would continue to use in the future is to remember that modern solutions like using the Cloud exist, but consider the trade-offs between the cloud-based solutions and on-premises servers.
